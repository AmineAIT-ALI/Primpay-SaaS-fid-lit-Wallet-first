Les 3 histoires racontées par le schéma (clique "Raconter une histoire" plus haut)

  1. ① Client ajoute sa carte — Marc scanne le QR public (flèche 1) → landing publique appelle l'API (2) → le module Customers crée l'identité (3) → demande à Wallet Mgr un pass (4) → Wallet Mgr génère le .pkpass via Apple PassKit ou Google Wallet (5) → le pass s'installe dans le Wallet du téléphone.
  2. ② Staff crédite un passage — Ali ouvre la PWA et scanne le QR du client (6) → API Gateway route vers Loyalty Engine (7) → Loyalty lit le programme du merchant (8) → vérifie cooldown et limite dans Redis (9) → écrit l'event dans PostgreSQL (10) → demande à Wallet Mgr d'actualiser le solde du pass (11) → Apple/Google pushent la mise à jour (12) → le nouveau solde apparaît dans le Wallet du client (13).
  3. ③ Reward consommée — même chemin 6→7, mais via POST /loyalty/redeem → Loyalty pose un lock transactionnel dans Redis (9) pour empêcher la double consommation → écrit l'event `redeem` (10) → met à jour le pass (11→13).

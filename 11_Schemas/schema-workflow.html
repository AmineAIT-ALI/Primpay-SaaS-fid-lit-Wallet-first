Workflow ① — Client ajoute sa carte fidélité
De "sticker QR aperçu sur le comptoir" à "pass installé dans le Wallet"
Contexte
  • Marc mange sa pizza
  • Voit un sticker "Ajoutez votre carte fidélité"
  • Sticker généré par nous, posé à l'install
1
📱
Client scanne le QR public
Déclencheur humain · appareil photo natif
Le sticker contient une URL signée : primpay.io/m/pizzeria-ali?s=sig
Sortie
  • URL ouverte dans le navigateur
  • Paramètre merchant_slug transmis
  • Signature HMAC vérifiée côté serveur
navigation HTTPS
Techno
  • Next.js statique (SSG)
  • Servi via Cloudflare
  • Pas de JS lourd, charge en < 1 s
2
🌐
Page d'ajout publique
primpay.io/m/pizzeria-ali
Page dédiée au merchant. Affiche son logo, son programme, un formulaire en 2 champs.
Demandé au client
  • first_name (texte)
  • phone (FR validé)
  • Case consentement RGPD cochée
POST /customers
Règles serveur
  • Vérifie signature du slug
  • Si phone existe déjà → réutilise le customer
  • Sinon → crée nouveau customer
  • Crée loyalty_account lié au merchant
3
⚙️
Module CUSTOMERS
POST /customers · NestJS
Crée l'identité globale ou retrouve l'existante. Crée un loyalty_account si premier contact avec ce merchant.
Écrit en DB
  • customers (si nouveau)
  • loyalty_accounts (toujours)
  • loyalty_events type onboarding
customer créé → generate pass
Détection device
  • User-Agent iOS → Apple
  • User-Agent Android → Google
  • Desktop → QR à scanner avec mobile
4
🎟️
Module WALLET MGR
Génère le pass · Apple PassKit + Google API
Génère le qr_payload + qr_signature HMAC, construit le pass (.pkpass ou JWT), l'envoie au client.
Contenu du pass
  • Logo merchant
  • Prénom + public_id (ex. AMIN-1423)
  • Progression "0 / 10 tampons"
  • QR fixe signé (payload + HMAC)
Apple PassKit API ou Google Wallet API
Infra externe
  • Apple : compte dev 99$/an
  • Google : gratuit
  • Latence 300-800 ms
5
🍎🤖
Apple / Google Wallet
Services tiers · génération du pass natif
Valide la signature, renvoie le fichier .pkpass (iOS) ou un JWT "Save to Google Wallet" (Android).
Retour
  • iOS : fichier binaire .pkpass
  • Android : URL save?jwt=...
  • Taille ~50 ko
pass renvoyé au téléphone
UX finale
  • Bouton "Ajouter à Apple Wallet"
  • ou "Enregistrer sur Google Wallet"
  • Clic → pass s'installe
6
✅
Pass installé dans le Wallet
Fin du flux · temps total < 30 s
Le client voit sa carte dans son Wallet. À la prochaine visite, il la présentera au staff. Un email de bienvenue est optionnel (post-MVP).
État après
  • wallet_passes.status = active
  • Compte prêt à être crédité
  • Le merchant apparaît "Nouveau client inscrit" dans son dashboard
Workflow ② — Staff crédite un passage
Du geste staff au pass mis à jour dans le Wallet du client (objectif < 2 secondes)
3 chemins possibles
  • SCAN du QR du pass
  • RECHERCHE prénom/téléphone
  • DERNIERS CLIENTS (clic direct)
1
👨‍🍳
Staff identifie le client
PWA ouverte sur téléphone/tablette
Ali sélectionne le client via l'un des 3 chemins. La PWA dispose toujours du JWT staff en mémoire (stocké après login).
Données côté PWA
  • customer_id identifié
  • merchant_id depuis JWT
  • staff_user_id depuis JWT
tap sur bouton "+1"
Headers
  • Authorization: Bearer <jwt>
  • Idempotency-Key: uuid
  • Content-Type: application/json
2
🚪
POST /loyalty/credit
API Gateway · Nginx + middleware NestJS
Vérifie JWT, extrait merchant_id, applique rate-limit, passe la requête au Loyalty Engine.
Payload
  • customer_id
  • source: "qr_scan"
  • points_delta: 0
  • stamps_delta: 1
route vers Loyalty Engine
Règles métier critiques
  • Cooldown < 10 min → REJETÉ
  • Limite jour atteinte → REJETÉ
  • QR révoqué → REJETÉ
  • Programme inactif → REJETÉ
3
⭐
LOYALTY ENGINE (cœur)
Valide toutes les règles métier
Récupère le programme actif, vérifie cooldown (Redis), limite jour (Redis), calcule le nouveau solde, détecte si un seuil est atteint.
Lectures
  • Postgres : loyalty_programs
  • Postgres : loyalty_accounts (balance)
  • Redis : cooldown:{m}:{c}
  • Redis : daily:{c}:{date}
règles OK → écrit en DB
Append-only
  • Jamais de DELETE
  • Jamais d'UPDATE d'event
  • Annulation = event compensatoire
4
🗄️
PostgreSQL — transaction
INSERT event + UPDATE balance atomique
Transaction : insère loyalty_events + met à jour loyalty_accounts.stamps_balance + crée une reward si seuil atteint.
Écrit
  • loyalty_events +1 row
  • loyalty_accounts.stamps_balance +1
  • rewards +1 si seuil atteint
  • Redis : set cooldown TTL
demande update pass
Contenu à mettre à jour
  • Nouveau solde "7 / 10"
  • Progression visuelle
  • Badge "Récompense dispo" si applicable
5
🎟️
WALLET MGR — update pass
Prépare le payload pour Apple/Google
Prépare la nouvelle version du pass, signe à nouveau, pousse vers Apple PassKit / Google Wallet.
Appel externe
  • Apple : POST /v1/devices/../passes
  • Google : PATCH loyalty/object/...
  • Async, sans bloquer la réponse staff
réponse immédiate au staff
UX staff
  • Animation verte
  • Chiffre qui défile 6 → 7
  • Vibration légère
  • Moins de 2 secondes total
6
✅
PWA affiche la confirmation
Retour à l'écran "Derniers clients"
Staff rend la main. Marc verra son pass mis à jour dans ~2-5 s via push Wallet (asynchrone, géré par Apple/Google).
Réponse JSON
  • event_id
  • new_balance: {stamps: 7}
  • progress: {current: 7, threshold: 10}
  • reward_available: false
push asynchrone vers Wallet client (Apple/Google)
Latence push
  • Apple APNs : 1-10 s
  • Google FCM : 1-5 s
  • Dépend du réseau du client
7
📲
Pass mis à jour chez le client
Apple Wallet / Google Wallet
Le pass du client affiche le nouveau solde la prochaine fois qu'il l'ouvre (ou immédiatement si notifications Wallet actives).
État final
  • Marc voit "7 / 10"
  • Event historique tracé
  • Ali peut passer au client suivant
Workflow ③ — Consommation d'une récompense
Le client a atteint le seuil. Le staff utilise la reward. Lock transactionnel pour éviter double consommation.
Déclencheur
  • Le client a 10/10 tampons
  • Reward déjà créée auto (flow précédent)
  • Status available en DB
1
🎁
Badge "Reward disponible" affiché
Fiche client dans la PWA staff
Le staff voit un badge coloré. Il confirme avec le client que c'est bien la pizza offerte aujourd'hui.
Données affichées
  • Libellé reward (ex. "1 pizza offerte")
  • Date d'obtention
  • Date d'expiration éventuelle
staff clique "Utiliser la récompense"
UX sécurisante
  • Modal obligatoire
  • "Offrir 1 pizza ?" OUI / NON
  • Évite les faux-clics
2
❓
Confirmation modale
PWA demande validation avant requête
Au clic OUI, la PWA envoie la requête. Un Idempotency-Key est généré côté client pour éviter les doubles envois.
Sortie
  • POST /rewards/:id/redeem
  • Idempotency-Key générée
POST /rewards/:id/redeem
Risque à contrer
  • Double-clic staff
  • Double tentative réseau
  • 2 staffs qui cliquent en même temps
  • Bug réseau partiel
3
🔒
LOCK transactionnel
Redis SETNX + Postgres SELECT FOR UPDATE
Acquiert un lock lock:reward:{id} sur Redis (5 s de TTL). Ouvre une transaction SQL avec SELECT ... FOR UPDATE sur la reward.
Si lock échoue
  • Retourne 423 Locked
  • PWA affiche "en cours de traitement"
  • Staff ne peut pas double-clic
lock obtenu
Vérifications
  • Reward toujours available ?
  • Non expirée ?
  • Toujours rattachée au bon merchant ?
4
✍️
PostgreSQL — COMMIT transaction
Change status + écrit event
UPDATE rewards SET status='redeemed', redeemed_at=NOW() + INSERT dans loyalty_events avec source reward_redeem.
Résultat
  • Reward consommée (irréversible)
  • Event tracé
  • Lock libéré
update pass Wallet (cycle recommence)
Cycle de fidélité
  • Compteur tampons remis à zéro
  • (ou au reliquat, selon règle)
  • Pass affiche "0 / 10"
5
🎟️
WALLET MGR — actualise le pass
Même mécanique que workflow ②
Le pass du client affiche le nouveau compteur. Il peut recommencer à accumuler. Le libellé "Récompense dispo" disparaît.
Push externe
  • Apple APNs ou Google FCM
  • Async
  • Latence 1-10 s
réponse staff immédiate
Feedback UX
  • Animation confetti légère
  • Message "Offert !"
  • Ligne "reward consommée" dans historique
6
🎉
Fin — reward consommée
Cycle de fidélité redémarre
Le merchant a servi sa récompense. Le client repart content. L'événement est tracé, comptabilisé dans les analytics du merchant (rewards consommées).
Impact analytics
  • "rewards_redeemed_30d +1"
  • Trackable dans dashboard owner
  • Indicateur de satisfaction
Workflow ④ — Signature + onboarding d'un commerçant
De la démo terrain au commerçant pleinement opérationnel
Contexte
  • Vito en démo chez Ali
  • Script terrain appliqué
  • Objections traitées
1
🤝
Accord verbal du commerçant
Face-à-face · < 3 minutes de démo
Ali dit "ok on essaie". Vito sort la tablette et démarre l'inscription en direct. Setup offerte si signature sous 48 h.
Levier commercial
  • Essai 14 jours sans CB
  • Setup offerte (sinon 99 €)
  • Sans engagement
Vito remplit le formulaire d'inscription
Infos collectées
  • Email owner
  • Mot de passe
  • Nom commerce
  • Adresse
  • Plan (Solo 29 € ou Multi 79 €)
2
📝
Création du compte merchant
POST /merchants + POST /staff_users (owner)
Crée merchants, merchant_locations (1 par défaut), staff_users (rôle owner), subscriptions en statut trialing.
Écrit en DB
  • merchants → status trial
  • staff_users → owner actif
  • subscriptions → trial 14 j
paramétrage du programme
À choisir avec le gérant
  • Points ou tampons ?
  • Seuil (ex. 10)
  • Libellé récompense
  • Cooldown (défaut 10 min)
3
⚙️
Configuration du programme
POST /merchant/programs
Insère un loyalty_program actif. Calcule les règles en JSON. Prêt à accueillir les premiers crédits.
JSON règles
  • type: "stamps"
  • rules: {"ratio": 1}
  • reward_policy: {threshold: 10, value: "1 pizza"}
création du QR public + génération du pack
Assets générés
  • QR public signé pour le merchant
  • PDF sticker prêt à imprimer
  • PDF chevalet
  • URL publique primpay.io/m/pizzeria-ali
4
🏷️
Génération du QR public + pack signalétique
Backend + job d'impression
Crée l'URL signée, génère un PDF imprimable immédiat pour tests. Commande du pack physique auto-déclenchée (expédition sous 3 jours).
Assets numériques
  • Stockés dans S3
  • Accessibles depuis dashboard
  • Re-téléchargeables à volonté
invitations staff
Staff ajouté
  • 1 à 3 emails saisis
  • Rôle staff par défaut
  • Email d'invitation envoyé
5
✉️
Envoi emails d'invitation
Resend · email transactionnel
Chaque staff reçoit un lien magique pour créer son mot de passe et installer la PWA. Validité 7 jours.
Contenu email
  • Sujet "Bienvenue sur Primpay"
  • Lien magique d'activation
  • Guide installation PWA
test en direct avec Vito
Check-list test
  • QR public scanne correctement
  • Ajout Wallet fonctionne (iOS et Android)
  • Staff peut scanner un pass
  • Crédit s'enregistre

🧪
Test en direct sur place
Vito simule un client réel
Vito scanne le QR public, ajoute une carte de démo, l'envoie chez lui. Ali passe côté staff et crédite. Validation des 2 chemins.

suivi automatisé (asynchrone)
Communications auto
